Remote & Multi-Client

One @zuse/server driven by the desktop, a mobile app, a browser, and remote / cloud dev-boxes — over WebSocket, reachable from anywhere, with gap-free reconnect and offline support.

Status: In progress Started: 2026-06-30 Foundation landed: PR0 + WS transport Companion: README.md, parallel-handoffs.md

Executive summary

Zuse today is a single Electron process: the renderer talks to @zuse/server over Electron IPC, chats live in SQLite as mutable CRUD rows, and live updates come from an in-memory pub/sub. We want the same server to be driven by many clients (desktop, mobile, browser) running on many hosts (laptop, SSH dev-box, cloud container) — all over one WebSocket contract.

The central insight

This is additive, not a rewrite. @zuse/server is already an Effect app on @effect/rpc / @effect/sql / @effect/platform-node. ADR 0007 already forbids electron imports in apps/server, mandates a pure makeMainLayer(deps) factory, and makes the transport pluggable. The base server stays. We add a WebSocket transport, an event-sourced sync core, a cloud relay, an Expo client, and SSH launch — each an independent track.

The single genuine architectural gap is persistence: mutable CRUD + in-memory pub/sub cannot give a mobile client gap-free reconnect, offline snapshots, or multi-writer ordering. We close it with event sourcing (an append-only log with a global sequence, projected into the existing tables) and, in the same pass, swap the SQLite driver to Node's built-in node:sqlite so one persistence layer runs identically under Electron and headless Node.

1 · Where we are today

Process model

Everything runs in the Electron main process. apps/server is imported in-process; the renderer (React 19 + Vite, Zustand for ephemeral state) is a thin client. There is no separate server process, no HTTP listener, no WebSocket — yet. The monorepo is Bun workspaces + Turbo:

PackageRole
apps/desktopElectron main + preload; builds with tsdown → dist-electron/*.cjs; owns the Electron-specific glue (windows, menus, the IPC transport).
apps/server (@zuse/server)All backend logic as a transport-agnostic Effect library: persistence, providers/drivers, git, pty, worktrees, permissions, credentials, code-index bridge, RPC handlers.
apps/rendererReact UI; consumes the RPC client; no durable local state.
packages/wire (@zuse/wire)The entire RPC surface (~100 methods) as @effect/rpc + Schema definitions. Transport-agnostic.
apps/web, apps/mcp-server, packages/index, …Marketing site, MCP transport for the index engine, code-index service, etc.

Transport & RPC

The contract lives in packages/wire as one MemoizeRpcs = RpcGroup.make(...). The transport is a RpcServer.Protocol implementation. Today the only implementation is Electron IPC:

renderer (RpcClient)
   │  electronClientProtocolLayer(bridge.rpc)  ── window.zuse bridge
   ▼  ipcRenderer.send(IPC_CHANNEL, frame)
Electron main
   │  electronServerProtocolLayer(webContents) implements RpcServer.Protocol
   ▼  RpcServer.layer(MemoizeRpcs) ← Handlers ← services/drivers (in-process)

Crucially, runtime.ts already takes the transport as an injected dependency:

export interface MainLayerDeps {
  readonly userData: string;                          // where sqlite/keychain live
  readonly folderPicker: typeof FolderPicker.Service; // host "choose folder" dialog
  readonly serverProtocol: Layer.Layer<RpcServer.Protocol>;  // ← the transport seam
}
// ServerLayer = RpcServer.layer(MemoizeRpcs) |> provide(Handlers) |> provide(deps.serverProtocol)

So a WebSocket transport is a new Layer<RpcServer.Protocol> handed to the same factory — no change to any service.

Persistence & data model

Storage is SQLite at <userData>/zuse.sqlite via @effect/sql-sqlite-node (better-sqlite3). Queries are raw @effect/sql template strings (no ORM); migrations are numbered Effect SqliteMigrator files run on boot. The model is direct, mutable CRUD:

TablePurpose
projectsWorkspace roots.
chatsSidebar entries; own 1+ sessions; read/unread tracking.
sessionsA thread with a provider/model, status, runtime/permission mode, resume cursor, optional worktree.
messagesOne row per renderable item: role, kind (= content _tag), content_json, parent_item_id, created_at.
queued_messagesPending user turns between sends. Created via lazy runtime DDL, outside the migrator.
worktreesmemoize-owned git worktree checkouts.
permissions, repository_settings, app_stateTool rules, per-repo overrides, key/value app state.

A message's content_json is a tagged union — user / user_rich / assistant / thinking / tool_use / tool_result / usage / context_usage / subagent_summary / user_question / error / interrupted and more.

Streaming today: streamMessages subscribes to a per-session in-memory PubSub<Message> before reading all rows ORDER BY created_at, builds a seen Set, and concatenates backfill + live (filtering dupes). Every reconnect replays the whole table and dedups in memory. Ordering relies on created_at, which is not monotonic within a millisecond.

2 · Where we're going

The environment model

The organizing idea of the end-state is the environment: "a host running @zuse/server." It is a first-class entity with a providerKind and an endpoint. The exact same server binary backs all three kinds:

providerKindWhere the server runsHow clients reach it
"desktop"The user's laptopElectron IPC in-process; or WS + managed tunnel for remote reach
"ssh"A remote dev-box, launched by the desktopSSH -L tunnel to 127.0.0.1:<local>
"cloud"A cloud container + persistent volumeRelay-provisioned managed endpoint (= cloud worktrees)
Why this matters

Everything above the transport — persistence, projector, providers, git worktree management — is host-agnostic (ADR 0007; worktrees are on-disk checkouts under the project path). So a cloud worktree is not a new code path: it is the headless server running on a cloud volume, registered with providerKind:"cloud", reached through the same link/connect + WS flow as desktop or SSH. We get cloud by holding four rules from day one: nothing in apps/server assumes a local machine; the relay keys by environmentId; connect/link methods carry providerKind + endpoint; worktree lifecycle stays a server-side RPC capability.

Four process shapes, all from one makeMainLayer(deps):

  1. In-process (today) — Electron; serverProtocol = electronServerProtocolLayer. Never stops working.
  2. Headless WS — bin.ts / zuse serve; serverProtocol = wsServerProtocolLayer({port}), file-backed AppPaths, no-op FolderPicker. landed
  3. Remote-over-SSH — shape 2 launched on a remote box, tunneled back.
  4. Cloud-hosted — shape 2 on a cloud container + volume, reached via the relay's managed endpoint.

A day in the life

Mobile on the LAN

Open the desktop → it shows a QR. Phone scans zuse://?pairingUrl=<host>#token=<code>, stores host+bearer in secure storage, opens a WS straight to the laptop, streams the chat from sinceSequence.

Mobile from anywhere

The phone (WorkOS-signed) links the environment to the relay with a DPoP key; the relay returns a managed Cloudflare endpoint + short-lived token; the phone connects directly to the laptop through the tunnel. The relay never sees chat bytes.

SSH dev-box

Pick a host from ~/.ssh/config; the desktop runs zuse serve there over SSH, opens a -L tunnel, and the renderer talks to ws://127.0.0.1:<local>/rpc as if local.

Cloud worktree

A provisioner boots zuse serve in a container with a persistent volume, clones the repo / creates the worktree on it, registers providerKind:"cloud". Identical UI, identical sync — the laptop can be asleep.

3 · Decisions

Each decision is recorded ADR-style: context, the call, alternatives weighed, and consequences.

D1 Event sourcing for chat persistence

Context

Mobile needs gap-free reconnect, offline snapshots, and multi-writer ordering. Mutable CRUD + in-memory pub/sub gives none of these.

Decision

Introduce an append-only events table with a global monotonic sequence in front of the existing tables. A projector derives messages/sessions/chats from events, so those tables keep their exact shape and every reader + the renderer stay essentially unchanged. messages effectively becomes a projection. Clients resume from a sinceSequence cursor.

Alternatives

Cursor-sync (rejected): add a sequence column + change-feed to the CRUD tables without an event log. ~70% of the value for less churn, but keeps mutable truth — no clean replay, audit, or rebuild — and forces a bespoke mobile sync layer. Not worth it for a multi-year foundation.

Consequences

(+) Single source of truth; deterministic rebuild ("drop messages, replay from 0"); trivial resume; aligns with the turn/checkpoint model. (−) Upfront refactor of message-store.ts; the discipline that all writes go through events (no in-place row mutation). Contained by keeping projections identical to today's tables and routing only sessions/messages through events in v1 (the queue stays CRUD — see R2).

D2 Persistence driver → node:sqlite

Context

The headless server must run off-laptop. Booting bin.ts surfaced two hard facts about the current driver (@effect/sql-sqlite-node = better-sqlite3, a native addon):

  • Bun can't load it at all — better-sqlite3 is not yet supported in Bun.
  • Native ABI is runtime-locked — the Electron prebuild is NODE_MODULE_VERSION 130; plain system Node is 131 and rejects it. A headless/SSH/cloud server cannot reuse the Electron binding.

Decision

Migrate persistence to Node's built-in node:sqlite (stable-unflagged from Node 22.13 / 23.4). No native addon → no ABI split → the same persistence code runs identically under Electron's Node and a headless Node. Its synchronous API is close to better-sqlite3, so the migration is mechanical. Do it inside the event-sourcing PR (D1) — both touch the persistence layer, so we edit message-store.ts and the migrator once.

Alternatives

Ship a Node-ABI better-sqlite3 binding for headless (fallback): keep two code paths and two prebuilt bindings. Strictly worse; reserved only for the case below. bun:sqlite: would couple the headless runtime to Bun and diverge from Electron's Node — rejected.

Consequences / prerequisite

Verify first

Confirm Electron 42's bundled Node exposes node:sqlite (it ships Node 22+, so it should). If it does not, fall back to the Node-ABI binding for headless and keep better-sqlite3 in Electron. @effect/sql's SqlClient interface + template queries are driver-agnostic, so migrations and repositories carry over; the work is a thin SqlClient over node:sqlite.

D3 Environment abstraction (providerKind)

Decision

Model "a host running the server" as an environment with providerKind: "desktop" | "ssh" | "cloud" and an endpoint { httpBaseUrl, wsBaseUrl }. Carry it through the connect/link wire methods from day one (done in PR0).

Consequences

Cloud-hosted worktrees become a provisioner bolt-on rather than a rearchitecture (see §2). The four design rules (no laptop assumption; relay keys by environmentId; methods carry providerKind; worktree ops stay server-side RPC) are the only cost paid now.

D4 Cloud reach: WorkOS + thin relay + managed tunnel

Decision

A control-plane relay links devices↔environments, issues short-lived connect tokens, and provisions a Cloudflare managed tunnel — then steps aside. Chat traffic is always direct client↔environment over WS; the relay is never in the data path. Identity reuses the existing WorkOS login (PKCE/keychain, getAccessToken seam), with a per-device DPoP proof-of-possession key. The environment proves control during linking by signing the relay's challenge with its local bearer token.

Alternatives

A new IdP (Clerk, etc.) — rejected; we already run WorkOS. Relaying chat data through the cloud — rejected (privacy, cost, latency, scaling).

Consequences

(+) Reach-anywhere with minimal trust surface; relay stays small and stateless-ish. (−) A new deployable + a new attack surface (mitigated: short-lived DPoP tokens, WorkOS-gated, no chat data — see R8).

D5 SSH remote dev-boxes: native ssh wrapper

Decision

Wrap the native ssh binary (no ssh2 library): ssh -G to resolve ~/.ssh/config; BatchMode=yes key auth with an SSH_ASKPASS password fallback; -N -L 127.0.0.1:local:127.0.0.1:remote with ServerAliveInterval; Effect Scope finalizers for teardown. A remote launch script runs zuse serve, picks a port, probes readiness, and returns {remotePort, serverKind}.

Consequences

(+) Uses what users already have (keys, agents, config); no native crypto deps. (−) Shells out; must parse ssh output and handle password prompts carefully.

D6 WebSocket transport via @effect/rpc socket protocol landed

Decision

Server: RpcServer.layerProtocolSocketServer over NodeSocketServer.layerWebSocket, JSON framing (matching the Electron path), orDie'd into MainLayerDeps.serverProtocol. Client: RpcClient.layerProtocolSocket over Socket.layerWebSocket. We do not hand-roll the protocol the way the Electron transport must — the framework gives per-connection clientId + automatic disconnect on socket close.

Consequences

The byte-for-byte same wire contract works across IPC and WS; multi-client fan-out comes for free (each socket is a distinct client). Headless bin.ts boots via NodeRuntime.runMain.

D7 Auth model: LAN bearer + cloud DPoP

Decision

Two tiers. LAN: the environment mints a bearer token shown as a QR; a paired client sends it as Authorization: Bearer over WS. Cloud: the user authenticates to the relay with WorkOS; the device holds a DPoP keypair; linking binds device→environment by having the environment sign a relay challenge with its local bearer; thereafter the relay issues short-lived DPoP-bound connect tokens for the managed endpoint.

Consequences

No long-lived secrets on the device; revocation is per-device at the relay; the LAN path works with zero cloud dependency.

5 · Core design

5.1 Events table + projector

CREATE TABLE events (
  sequence       INTEGER PRIMARY KEY AUTOINCREMENT,  -- global monotone cursor
  event_id       TEXT NOT NULL UNIQUE,               -- UUID, idempotency
  stream_kind    TEXT NOT NULL,                      -- 'session' | 'chat' | 'queue'
  stream_id      TEXT NOT NULL,
  stream_version INTEGER NOT NULL,                   -- per-stream 1..N (optimistic lock)
  type           TEXT NOT NULL,                      -- 'MessagePersisted' | 'StatusChanged' ...
  occurred_at    TEXT NOT NULL,
  actor          TEXT,                               -- clientId / 'system'
  payload_json   TEXT NOT NULL
);
CREATE INDEX        idx_events_stream         ON events(stream_kind, stream_id, sequence);
CREATE UNIQUE INDEX idx_events_stream_version ON events(stream_kind, stream_id, stream_version);
ALTER TABLE messages ADD COLUMN sequence INTEGER;     -- messages becomes a projection
-- backfill: one synthetic MessagePersisted per existing row in (created_at, rowid) order,
-- stamping the assigned sequence onto both the event and messages.sequence.

appendEvent inserts the event and projects in the same transaction, retrying on the unique-version conflict; persistMessage keeps its signature and returns the assigned sequence:

const appendEvent = (p) => sql.withTransaction(Effect.gen(function* () {
  const v = (yield* sql`SELECT MAX(stream_version) AS v FROM events
                        WHERE stream_kind=${p.streamKind} AND stream_id=${p.streamId}`)[0]?.v ?? 0;
  const [{ sequence }] = yield* sql`INSERT INTO events (...) VALUES (..., ${v + 1}, ...)
                                    RETURNING sequence`;
  yield* projectEvent({ sequence, ...p });   // for MessagePersisted = today's messages INSERT
  return { sequence };
})).pipe(Effect.retry({ while: isStreamVersionConflict, times: 5 }), Effect.orDie);

A boot-time ProjectorCatchup layer replays events past max(messages.sequence), giving deterministic rebuild for free. v1 routes only MessagePersisted through events; lifecycle UPDATEs (status/cursor) may stay inline and migrate later. The queue stays CRUD (R2).

5.2 Cursor streaming & the reconnect protocol

The wire already has MessageEnvelope { sequence, message } and optional sinceSequence (PR0). The new stream subscribes before reading, replays only the delta, and live-tails past the last replayed sequence — no seen Set, O(1) memory:

const streamMessages = (sessionId, sinceSequence) => Stream.unwrapScoped(Effect.gen(function* () {
  const since   = sinceSequence ?? 0;
  const dequeue = yield* (yield* getOrMakePubsub(sessionId)).subscribe;  // subscribe FIRST
  const rows    = yield* sql`SELECT ... FROM messages
                             WHERE session_id=${sessionId} AND sequence > ${since}
                             ORDER BY sequence ASC`;
  const replay  = rows.map(r => MessageEnvelope.make({ sequence: r.sequence, message: fromRow(r) }));
  const last    = rows.at(-1)?.sequence ?? since;
  const live    = Stream.fromQueue(dequeue).pipe(Stream.filter(e => e.sequence > last));
  return Stream.concat(Stream.fromIterable(replay), live);
}));
client            environment
  │  stream(session, sinceSequence=42)   │
  │ ───────────────────────────────────► │  SELECT WHERE sequence > 42
  │ ◄─── env{43}, env{44}, … (replay) ─── │
  │ ◄─── env{45} live … (tail) ────────── │  broadcast AFTER commit
  ✗  (network drops at 47)                │
  │  stream(session, sinceSequence=47)    │  ← resume: only the delta, gap-free
  │ ───────────────────────────────────► │

The renderer records the highest sequence per session and passes it back on resubscribe. broadcastMessage publishes the envelope after commit — a subscriber must never observe an event that later rolls back; the cursor resume closes any crash-between-commit-and-publish gap.

5.5 SSH launch & tunnel lifecycle

ssh -G host resolves config → ssh user@host sh -s -- <stateKey> pipes a launch script that picks a free port, detects Node, runs zuse serve on 127.0.0.1:<port> under ~/.zuse, probes HTTP readiness, prints {remotePort, serverKind} → the desktop opens -N -L 127.0.0.1:<local>:127.0.0.1:<remote> and hands the renderer ws://127.0.0.1:<local>/rpc. An Effect Scope finalizer tears down the forward (SIGTERM→SIGKILL) and stops the remote server on disconnect.

5.6 Mobile: offline & reconcile

The Expo client imports MemoizeRpcs from @zuse/wire directly (the contract is transport-agnostic — no type duplication). It caches shell/thread snapshots as JSON in expo-file-system and the connection catalog (host + bearer, keyed by environmentId) in expo-secure-store. On launch it renders the snapshot instantly, then opens the stream with the snapshot's highest sequence as sinceSequence and reconciles the delta.

8 · Risks & mitigations

#RiskMitigation
R1Partial-streaming assistant rows could break append-only if a driver mutates a row in place.Drivers emit deltas as discrete full rows (eventToContent) today; forbid in-place mutation (use a MessageSuperseded event if ever needed). Audit callers.
R2queued_messages uses lazy DDL outside the migrator — races a transactional projector.Keep queue as CRUD in v1; fold its DDL into a real migration; don't route it through events.
R3Multi-client concurrent appends collide on stream_version.appendEvent bounded-retries recomputing MAX+1; the unique index enforces order.
R4Existing rows have no originating event.Backfill in the events migration, (created_at, rowid) order, before any client connects (migrator runs upstream of MessageStore).
R5Event committed but broadcast lost on crash.Append+project in one txn; broadcast after commit; sinceSequence resume reads the committed row.
R6messages.stream success type flip is a typed flag-day.Single repo, ~1 consumer; grep before landing PR-A2.
R7Electron's bundled Node may predate node:sqlite.Verify first (D2 prerequisite); documented better-sqlite3 Node-ABI fallback for headless.
R8Relay is a new attack surface.Short-lived DPoP tokens, WorkOS-gated, no chat data in path; pen-test before exposing.

9 · Verification & rollout

  • A — appendEvent ordering + unique-violation retry tests; property test "replay from 0 reproduces the messages projection"; reconnect test (stream → drop → resubscribe sinceSequence → zero gaps/dupes), in-memory sqlite. Electron app still streams a chat end-to-end.
  • B — boot zuse serve, point the renderer at ws://127.0.0.1:<port>/rpc, run a chat; confirm IPC path unaffected.
  • G — ssh -G resolve a fixture host; launch + tunnel + drive a chat on a real box; kill tunnel → assert Scope teardown.
  • F — Expo on LAN, QR-pair, view a live chat, send + approve; airplane-mode → snapshot renders → reconnect reconciles.
  • D/E — link a stub environment, fetch a connect token, reach the desktop through the tunnel from cellular; assert no chat bytes traverse the relay.
  • H — trigger an approval; assert APNs delivery + Live Activity update.
